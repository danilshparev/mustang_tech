# mustang_tech
# Проектирование базы данных и автоматизация бизнес-логики

## Общее описание проекта
Проект посвящен разработке реляционной структуры базы данных и логики автоматизации на уровне СУБД для компании «Мустанг — Технологии кормления» (крупный производитель кормов, заменителей цельного молока и премиксов). 

Основная цель работы — создание отказоустойчивой архитектуры данных для ERP/SCM-системы, которая обеспечивает строгую целостность данных при закупках сырья и продажах готовой продукции, а также автоматизирует контроль ветеринарных рисков и коммерческих показателей.

## Системный анализ и структура данных
В ходе анализа предметной области база данных была приведена к третьей нормальной форме, что исключает избыточность данных и аномалии модификации. Структура состоит из следующих логических слоев:
* **Управление поставками (SCM):** таблицы регионов (`regions`), поставщиков (`supplier`) и сырья (`raw_material`).
* **Производство и склад (ERP):** учет заказов на закупку (`purchase_order`), номенклатура готовой продукции (`finished_good`) и остатки на складах (`inventory`).
* **Дистрибуция и продажи:** учет клиентов (`customer`) и заказов на реализацию (`sales_order`).
* **Мониторинг рисков:** регистрация внешних санитарно-ветеринарных событий (`external_event`).

### Обеспечение целостности данных (Data Integrity)
Для минимизации ошибок ввода и защиты данных на уровне схемы внедрены строгие доменные ограничения (Check Constraints) с использованием регулярных выражений:
* Проверка корректности формата телефонных номеров контрагентов: `CHECK (phone ~ '^\+7\(\d{3}\)\d{3}-\d{2}-\d{2}$')`.
* Валидация маски артикулов готовой продукции в зависимости от продуктовой категории: `CHECK (product_code ~ '^(ZCM|PRM)-\d{4}$')` (где ZCM — заменители цельного молока, PRM — премиксы).

## Реализованная бизнес-логика и автоматизация

### 1. Представления (Views) для аналитической отчетности
В проекте реализованы SQL-представления, позволяющие отслеживать операционные показатели без создания избыточных таблиц:
* Вывод актуальных остатков сырья на складах для оперативного планирования закупок.
* Анализ маржинальности продаж готовой продукции в разрезе менеджеров и типов клиентов для контроля коммерческой эффективности.

### 2. Пользовательские функции и процедуры
Разработана логика контроля ценообразования. Функция перед оформлением сделки проверяет соответствие продажной стоимости минимально допустимому уровню рентабельности продукта (`min_margin_pct`). При попытке продать товар ниже себестоимости с учетом минимальной маржи СУБД блокирует транзакцию.

### 3. Триггеры для управления рисками
Реализован механизм автоматического реагирования на изменение ветеринарной обстановки. При регистрации критического внешнего события (например, вспышки заболеваний в регионе) триггер `trg_set_region_risk_zone` автоматически переводит регион в статус «Зона риска». Это позволяет на уровне СУБД ограничить или заблокировать закупки сырья у поставщиков, находящихся в опасной зоне.

### 4.Оконные функции
Для формирования сквозной бизнес-аналитики применены оконные функции PostgreSQL:
* Расчет средней рыночной цены на сырье по всем утвержденным заказам с помощью `AVG(...) OVER (PARTITION BY material_id)` для оценки эффективности закупочных цен.
* Ранжирование и сквозной учет причин отмен заказов клиентами через функцию `ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date)` для выявления проблем в сервисе или качестве продукции.
